<p>Debugging und Logging Code strotzt meist vor Funktionsnamen und co. – nur damit man am Ende im Log wieder die passende Stelle im Code findet. Natürlich gibt es noch weitere Gründe herauszufinden wer welchen Code aufgerufen hat. Das Interface <a href="http://msdn.microsoft.com/en-us/library/system.componentmodel.inotifypropertychanged.aspx">INotifyPropertyChanged</a> ist auch solch ein Beispiel. Hier wird der Name des Properties benötigt.</p> <p>Dies kann man natürlich statisch lösen – allerdings geht dies beim Refactoring meist schief. Mit .NET 4.5 kommt eine nette Hilfe dazu. Vermutlich kennen es schon einige – aber ich bin erst vor kurzem drüber gestolpert.</p> <h3>Caller Information</h3> <p>In .NET 4.5 ist die Möglichkeit hinzugekommen über 3 Attribute Parameter dekorieren zu können. Diese Parameter werden während des Kompilierens mit den “Aufruf”-Informationen befüllt.</p> <p><a href="http://msdn.microsoft.com/en-us/library/system.runtime.compilerservices.callermembernameattribute.aspx"><strong>CallerMemberName</strong></a>: Gibt den Namen der aufrufenden Methode oder des Properties wieder.<br><a href="http://msdn.microsoft.com/en-us/library/system.runtime.compilerservices.callerfilepathattribute.aspx"><strong>CallerFilePath</strong></a>: Gibt den vollen Dateipfad (der beim Kompilieren genutzt wurde) wieder.<br><a href="http://msdn.microsoft.com/en-us/library/system.runtime.compilerservices.callerlinenumberattribute.aspx"><strong>CallerLineNumber</strong></a>: Die Zeilennummer des aufrufenden Codes.</p> <p>Die Attribute befinden sich alle im <a href="http://msdn.microsoft.com/en-us/library/System.Runtime.CompilerServices.aspx">System.Runtime.CompilerServices</a> Namespace.</p> <h3>Beispiel</h3> <div id="codeSnippetWrapper" style="overflow: auto;
